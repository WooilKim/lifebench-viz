다음 이벤트 데이터와 사용자 프로파일 데이터를 기반으로 {num_questions}개의 질문을 생성하세요.
두 가지 대카테고리: 1) 사용자 모델링 질문 (패턴 인식 + 사용자 모델링), 2) 개방형 지식 추론 질문.

이벤트 데이터:
{event_info}

사용자 프로파일 데이터:
{persona_info}

{num_questions}개의 질문을 생성하세요. 사용자 모델링 질문과 개방형 지식 추론 질문의 비율은 2:1, 절반은 주관식, 절반은 객관식.

**사용자 모델링 질문 (user_modeling)**: 패턴 인식과 사용자 모델링 두 가지 하위 질문 포함:
- **패턴 인식**: 여러 이벤트를 기반으로 개인의 행동 패턴, 습관, 선호, 성격 특성 또는 특정 일을 하는 일반적인 방식/선호도 식별.
  예: 특정인의 운동 습관, 독서 선호도 또는 사교 패턴 분석.
- **사용자 정보 인식**: 사용자 관련 정보 상황 질문. 예: 업무/생활 상황 (이 시간에 어디서 무슨 일을 하는지), 관계 상황 (XX는 누구인지) 등, 사용자 프로파일 구축 소급 지원.

**개방형 지식 추론 질문 (open_domain)**: 이벤트 정보와 상식 지식을 결합하여 추론. 알려진 사실을 기반으로 미지의 결론 추론.
예: 특정인이 《청화자》를 자주 듣는다면 주걸륜을 좋아할 가능성 추론, 또는 특정인이 자주 병원에 간다면 건강 문제가 있을 가능성 추론.

질문 설계 원칙:
1. W원칙(Who/What/When/Where/Why/How/If) 참조, 질문 유형에 맞는 차원 선택
2. 사용자 모델링 질문의 패턴 인식 부분은 활동 요약, 행동의 반복성/규칙성/일관성에 집중.
   예: "나는 보통 몇 시에 운동을 하나요?"
3. 개방형 지식 추론 질문은 이벤트 정보와 상식을 결합하여 합리적인 추론 체인 구축.
   예: "내가 듣는 음악 유형을 보면, 어떤 가수를 좋아할 가능성이 높나요?"
4. 질문 표현은 검색을 돕는 정보를 최소화하되 관련 이벤트를 특정할 수 있는 충분한 정보량 보장
5. 질문 설계 시 직접 답변을 추론할 수 없어야 하며, 여러 관련 이벤트를 분석해야 결론을 추론할 수 있어야 함.
   예: "청화자에 기반하여 주걸륜을 좋아하나요?"라고 직접 묻는 것이 아니라 "내가 듣는 음악을 보면 어떤 가수를 좋아할 가능성이 높나요?"라고 물어야 함
6. 시간 모호화 전략:
   - 질문 설명에 월(예: "2월", "3월")을 언급, 우선적으로 월 사용으로 시간 위치 파악
   - 이벤트가 다른 시간에도 발생할 수 있다면, 반드시 월(예: "2월")로 모호화
   - 같은 달에 동일 이벤트가 여러 번 발생할 수 있다면, 날짜 범위(예: "2월 5~10일")로 모호화
   - 입력 데이터 범위 내 혼동 유발 유사 이벤트가 없으면 원래 날짜 사용 가능
   - 위 방법으로도 정확한 위치 특정이 어려우면 구체적 날짜 명시
7. 이벤트 설명을 합리적으로 단순화 가능, 관련 이벤트 위치 파악에 충분한 핵심 정보만 보존
8. 등가 대체 가능. 예: 호칭을 이름으로 대체 ("딸"을 "소미"로 대체)

질문 설계 전략:

【사용자 모델링 질문 전략】
1. 행동 패턴 인식: 반복 발생하는 행동 시퀀스나 조합 식별. 예: "평일 활동 배치 패턴", 또는 어떤 일을 하는 습관적 선호 (예: XX 체육관 선호)
2. 습관 분석: 개인의 일상 습관과 고정 행동 분석. 예: "기상 후 습관적 활동"
3. 선호도 인식: 개인의 선호도와 성향 추론. 예: "좋아하는 음악 유형이나 영화 장르"
4. 특성 추출: 개인의 성격 특성이나 속성 정보 요약. 예: "건강 관리를 중시하는지 여부"
5. 규칙 발견: 행동 뒤에 있는 규칙과 주기성 발견. 예: "운동 활동의 빈도와 시간 규칙"
6. 업무/생활 상황: 특정 월의 근무지, 직업 또는 생활 상태 질문. 예: "나 3월에 주로 무슨 프로젝트를 했나요, 야근했나요?"
7. 프로파일 정보 선택: 이벤트 데이터에서 프로파일 정보를 반영하는 데이터 선택, 어려운 프로파일 정보 우선 선택. 예: 샤오홍의 직업은 무엇인가요, 아버지 5월 건강 상태는 어떤가요.

【개방형 지식 추론 질문 전략】
1. 연관 추론: 이벤트 간 연관 관계에 기반한 추론. 예: "구매한 책을 통해 독서 관심사 추론"
2. 상식 적용: 상식 지식을 결합하여 이벤트 설명. 상식 지식 포함: "프랑스 수도는 파리", "베토벤은 고전주의 음악가" 같은 통용 지식. 예: "프랑스 여행 이벤트를 통해 체류 가능 도시 추론"
3. 선호도 확장: 알려진 선호도에서 관련 선호도 추론. 예: "좋아하는 가수에서 다른 가수 추론" 또는 "음악 취향에서 베토벤 음악 적합성 추론"
4. 동기 추론: 행동에서 잠재적 동기 추론. 예: "주말 일정을 통해 휴식 방식 선호도 추론"

주의: 개방형 지식 추론은 설명과 질문을 수정하는 방식으로 구현 가능하며, 반드시 추론이 필요한 것은 아님. 예: 원래 이벤트가 파리 여행이라면, 질문을 "프랑스 수도 여행 시"로 변경하면 상식 지식 추론이 됨.

예시 질문:

【사용자 모델링 질문 (user_modeling)】
- 주관식: 나는 3월에 보통 평일 어떤 시간에 운동을 하나요?
- 객관식: 2월 회의 주요 장소는?
  A. 사무실  B. 회의실  C. 온라인  D. 고객 현장
- 주관식: 나는 5월에 주로 어디서 근무하나요?
- 객관식: 이명은 누구인가요?
  A. 동료  B. 가족  C. 친구  D. 고객

【개방형 지식 추론 질문 (open_domain)】
- 주관식: 내가 4월에 듣는 음악 유형을 보면, 어떤 가수를 좋아할 가능성이 높나요?
- 객관식: 내 8월 행동을 보면, 다음 중 나에 대해 추론할 수 있는 것은?
  A. 독서를 좋아함  B. 학술 연구 종사  C. 조용한 환경 추구  D. 이상 모두 해당 가능
- 주관식: 내가 프랑스 수도로 여행했을 때 며칠을 머물렀나요?
- 객관식: 내 8월 음악 취향을 보면, 베토벤이 적합한가요?
  A. 적합  B. 부적합  C. 판단 불가  D. 부분 적합

요구사항:
1. 질문은 이벤트 데이터 기반 필수. 사용자 모델링 질문은 행동 규칙이나 사용자 정보에 집중, 개방형 지식 추론 질문은 상식 결합
2. 질문 유형 비율:
   - user_modeling과 open_domain 비율 2:1
   - 객관식과 주관식 비율 1:1
3. 최소 정보 원칙 준수, 필요한 핵심 정보만 포함, 과도한 검색 힌트성 내용 금지
4. 반드시 정보 모호화 처리 사용: 시간 모호화, 이벤트 설명 단순화, 등가 대체 등
5. 질문 설계는 직접 답변 추론 금지, 여러 관련 이벤트 분석을 통해서만 결론 추론 가능, 이벤트 속성 직접 추출 금지
6. 질문은 일정 수준의 깊이와 복잡성 필요, 과도하게 단순한 직접 추출 금지
7. 답변은 구체적이고 명확, 개방형 또는 정확성 판단이 어려운 질문 금지
8. 객관식은 4개 선택지 제공, 정답은 1개
9. 객관식 선택지 내용은 최소 정보 원칙 준수, 필요한 핵심 정보만 포함, 추가 서술이나 추론 금지, 답변에 도움이 되는 정보 추가 금지

각 질문에 대해 제공:
- 구체적 질문 내용
- 구체적이고 명확한 답변
- 채점 기준 설계 (만점 10점), 각 채점 기준은 하나의 명확한 사실 정보에 대응
- 답변 도출에 필요한 구체적 이벤트 (전량 추출 불필요, 답변 추론에 충분한 이벤트 수만)
- 각 이벤트의 발생 날짜
- 질문 유형 (question_type): user_modeling 또는 open_domain
