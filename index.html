Step 0: Persona Generation

run/persona_gen.py → persona/persona_gen.py

실행 시간: ~5-10분/인물
LLM 호출: ~3-4회/인물
파이프라인 역할

실제 시뮬레이션에 사용될 실제적인 인물 프로파일 생성. 모든 이후 단계의 기반이 됨

핵심 입력

person.json (실제 수집 데이터 10명) + refer.json (속성 풀 럨덤 선택)

핵심 산출

persona.json — 상세 인물 프로파일 (30+ 필드, 인간관계망 포함)

⚠️ 주의

프롬프트는 전부 중국어로 작성되어 있어요. 데이터는 중국어로 생성된 후 영어 버전으로 종간 번역됩니다. 이후 모든 파이프라인 단계의 프롬프트도 중국어입니다.

Persona 생성 내부 흐름

실제 수집 데이터
data/person.json — 10명의 기본 데이터 (이름, 생년, 직업, 주소)
예: 冒浩然, 1994-10-29, 금융투자고문, 화타이증권
LLM PersonaGenerator.gen_profile()
persona/persona_gen.py → template + template_refine
IN: person.json 1명 + refer.json 랜덤 속성 (취미/가치관/목표)
OUT: 상세 페르소나 JSON (30+ 필드)
LLM PersonaGenerator.refine() — 검증/정제
persona/persona_gen.py → template_refine
IN: 생성된 페르소나 (1단계 결과)
OUT: 논리 일관성 검증 완료된 페르소나 (연령/직업/수입/주소 관계 조정)
LLM PersonaGenerator — 인간관계망 생성
persona/persona_gen.py → template_relation_1 + template_person
IN: 완성된 페르소나 + 관계 유형 (가족/동료/친구)
OUT: relation[] — 각 인물의 이름/연령/직업/주소/MBTI/연락처
persona.json 저장
각 사용자 폴더에 저장. 리파이너 단계에서 영어로 번역된 버전으로 교체
⚠️ 모든 프롬프트는 중국어로 작성되어 있어요. LifeBench가 중국어 기반 프로젝트이기 때문입니다.

person.json 수정 → 전혀 다른 인물
  • age를 45로 올리면 → 직업/수입/가족 구성이 달라짘
  • job을 "의사"로 매핑하면 → 병원 관련 이벤트 비율 증가
  • salary를 올리면 → 고급 레스토랑/건강클럽 이벤트 늘어남
refer.json 수정 → 속성 풀 확장
  • 취미 목록에 특이한 항목 추가 → 해당 취미 만나는 이벤트 생성 증가
  • 맥락 유형 다양화 → 더 다양한 생활 패턴
template_refine 수정 → 검증 기준 변경
  • 검증 엄격도 높이면 → 더 현실적인 페르소나 (but LLM 호출 늘어남)
  • 속성간 일관성 체크 항목 수정 → 가중치 변경 가능

Step 1: Draft Generation

연간 생활 이벤트 타임라인 초안 생성

실행 시간
~30-60분

🎯 목적

persona.json을 기반으로 2025년 1월 1일부터 12월 31일까지 365일간의 주요 생활 이벤트 타임라인을 생성합니다. 이 단계에서는 "큰 그림"만 그립니다 - 각 날짜의 개략적인 이벤트 목록과 간단한 설명만 포함됩니다.

⚙️ 실행 흐름

1
연간 주요 이벤트 생성 (100개)
genevent_yearterm() → 4개 프롬프트 순차 실행
2
중요 사건 추출
_extract_impact_events() → 인생 변화 이벤트만 필터링
3
월별 세부 이벤트 생성
_analyze_monthly_events() → 각 월마다 일별 이벤트 생성
4
타임라인 충돌 해소
optimize_merged_timelines() → 시간/공간적 모순 제거

📥 Input

persona.json (Step 0에서 생성)

📤 Output

daily_draft.json (365일 이벤트 목록)

📂 실행 파일 경로

메인 실행 파일
run/draft_gen.py
핵심 로직
event/scheduler.py

1. template_yearterm_eventgen (1단계: 100개 이벤트 생성) ▼
사용 함수: genevent_yearterm()
请基于下方的【人物分析】，为该人物推理生成 2025 年内可能发生的 100 件符合其画像的生活事件...

## 一、核心创作原则

### 1. 真实性与贴合性
- 深度贴合人物画像：所有事件必须与人物的职业背景、家庭关系、生活习惯、性格特质、经济状况、社交圈强关联
- 符合现实逻辑：事件需遵循时间规律（如"冬季滑雪""春节返乡"）
- 自然的因果关联：部分事件之间应存在合理的因果关系

### 2. 多样性与平衡性
- 全面覆盖生活维度：事件需涵盖职业、家庭、社交、健康、财务、个人成长等多个领域
- 事件类型多样化：包含短期事件（1-3天）、中期事件（1-4周）、长期事件（1-3个月）

#### （1）人群共性事件

- 生成"因他人需求而参与"的非日常事件

5. template_extract_impact_events (중요 사건 추출) ▼
사용 함수: _extract_impact_events()
请基于以下人物画像和事件列表，仅提取两类重要事件：

## 核心任务

### 第一类：画像信息产生显著改变的事件
判断标准：事件发生后必须需要更新人物画像的具体字段信息
具体类型：工作生活上大的变动、职业、教育里程碑、财务、健康变化、人际关系变化

### 第二类：影响近期生活习惯的事件
判断标准：事件会导致个人在特定时间段内改变作息安排、生活习惯
具体类型：短期出差、旅行、生病、连续加班、短期培训

严格排除的事件类型：
- 仅增进感情但未改变关系状态的社交活动
- 未导致生活习惯改变的短期情绪波动
- 24小时内可恢复的短暂不便

## 输出格式
对这段时间事件的总结分析
[

        "event_content": "事件描述",
        "original_event_id": "ID",
        "impact_category": "Career/Health/...",
        "impact_description": "具体影响",
        "duration": "持续时间"
    }}
]

각 생성기의 역할과 생성 규칙

1. CommunicationOperationGenerator (전화 + 문자) ▼
생성 규칙
  • 각 이벤트의 참여자와 전화/문자 매핑
  • 직장: 업무 관련 통화 (평균 5-10분)
  • 가족: 개인 통화 (평균 15-30분)
  • 친구: 짧은 통화 + 많은 문자
  • 문자 내용은 이벤트 컨텍스트 반영
Context 프롬프트
'你是一位手机数据专家和深度用户，深谙用户使用手机的习惯和行为模式'
2. FitnessHealthOperationGenerator (피트니스) ▼
생성 규칙
  • daily_event의 운동 이벤트 → 상세 피트니스 데이터 변환
  • 걸음수: 하루 평균 6000-12000보
  • 심박수: 휴식 시 60-80bpm, 운동 시 120-160bpm
  • 수면 데이터: 취침/기상 시간 + 수면 단계
  • 건강 상태(health_analysis)와 일관성 유지
3. ChatOperationGenerator (채팅) ▼
생성 규칙
  • 각 이벤트마다 자연스러운 대화 생성
  • 친구와 저녁 약속 → "오늘 7시에 어디서 만날까?" 스타일 대화
  • 가족 통화 예고 → "엄마, 오늘 저녁 통화할게요" 메시지
  • 이모티콘, 스티커 포함 (중국 WeChat 스타일)
4. PushOperationGenerator (푸시 알림) ▼
생성 규칙
  • 인물의 직업/관심사에 맞는 앱 알림
  • 금융 어드바이저 → 금융 뉴스, 주식 알림
  • 운동 좋아함 → 피트니스 앱 알림
  • 일반: 날씨, 택배, 소셜미디어 알림

💡 주요 특징

  • event_id 연결: 각 데이터가 어떤 이벤트에서 발생했는지 추적 가능

Phone Data 생성 수정 가이드

✏️ 통화 빈도 조절

CommunicationOperationGenerator 프롬프트에서
每天平均 3-5 通电话 → 숫자 변경
• 1-2통: 말이 적은 성격
• 8-10통: 사교적인 성격

✏️ 푸시 알림 밀도

PushOperationGenerator에서
낮음: 하루 5-10개 (알림 꺼둠)
보통: 하루 20-30개
높음: 하루 50-80개 (알림 중독)

💡 채팅 스타일 변경

ChatOperationGenerator context에서
• '你是一个正式礼貌的职场人' → 업무적 톤
• '你是一个活泼可爱的年轻人' → 이모티콘 많이

⚠️ 주의사항

• 7개 생성기가 병렬 실행되므로 각각 독립적으로 수정 가능
• event_id 연결이 깨지지 않도록 주의
• 사진은 메타데이터만 생성 (실제 이미지 생성 X)

Step 4: QA Generation

3가지 난이도 질의응답 데이터 생성

실행 시간
~20-30분

🎯 목적

생성된 모든 데이터 (daily_event + phone_data)를 기반으로 LLM 벤치마크용 질의응답 데이터를 생성합니다. 단순 사실 확인부터 복잡한 추론까지 3가지 난이도로 구성됩니다.

📊 QA 타입

Single-hop QA
100+ 문제
단일 데이터 소스에서 직접 답변 가능
예: "1월 1일 몇 시에 일어났나?"
Multi-hop QA
50+ 문제
2-3개 정보 연결 필요
예: "왜 에너지가 감소했나?"
Reasoning QA
20+ 문제
패턴 인식, 예측 필요
예: "내일 운동할 확률은?"

📥 Input

daily_event.json + phone_data/ (모든 파일)

📤 Output

QA/*.json (3개 파일)
1. Single-hop QA (단일 홉) ▼
특징: 하나의 데이터 소스에서 직접 답변 가능
카테고리:
  • factual_time: "몇 시에 ~ 했나?"
  • factual_person: "누구와 만났나?"
  • factual_location: "어디에 갔나?"
  • factual_activity: "무엇을 했나?"
예시:
Q: What time did Feng Haoran start his morning run on January 1st?
A: 08:00
Source: daily_event.json, event_id=2
2. Multi-hop QA (다중 홉) ▼
특징: 2-3개 정보를 연결해야 답변 가능
추론 타입:
  • temporal_reasoning: 시간 흐름 추론
  • causal_reasoning: 인과관계 추론
  • cross_modal: 여러 데이터 소스 통합
예시:
Q: Why did Feng's energy level decrease on January 3rd?
A: He stayed up late (睡觉时间: 00:00) due to work pressure, and had no exercise that day.
Reasoning Steps:
1. Check daily_event: 睡觉时间 = 00:00 (late bedtime)
2. Check events: "Random Event: Emotional Fluctuations Due to Work Pressure"
3. Check 今日运动: None
Sources: daily_event.json, daily_draft.json
3. Reasoning QA (추론) ▼
특징: 패턴 인식, 예측, 반사실적 추론 필요
추론 타입:
  • pattern_recognition: 습관 패턴 파악
  • prediction: 미래 행동 예측
  • counterfactual: "만약 ~했다면?" 추론
예시:
Q: Based on Feng's habits in January, what is the likelihood he will exercise on a rainy day?
A: Low. Analysis of January shows he consistently canceled outdoor runs on rainy days (Jan 4, 12, 23) and did not replace them with indoor activities.
Pattern Evidence:
- 3 rainy days in January → 0 outdoor exercise
- No gym membership or indoor exercise mentioned
- ENTJ personality: efficiency-focused, prefers planned activities

QA 생성 수정 가이드

✏️ QA 개수 조절

qa_generator.py에서
• single_hop: 100개 → 50개 (간단한 벤치마크)
• multi_hop: 50개 → 100개 (추론 중심 벤치마크)
• reasoning: 20개 → 50개 (고난이도 벤치마크)

✏️ 질문 스타일 변경

프롬프트에서
• "Generate natural questions" → 일상적 질문
• "Generate precise factual questions" → 정확한 사실 확인
• "Generate open-ended questions" → 서술형 답변

💡 난이도 밸런스

현재 비율: Single(60%) : Multi(30%) : Reasoning(10%)
고급 벤치마크: 30% : 40% : 30%
초급 벤치마크: 80% : 15% : 5%

⚡ 최종 목적

이 QA 데이터는 LLM 평가의 최종 목적물입니다.
RAG 시스템이나 에이전트의 "개인 생활 이해 능력"을 테스트합니다.
전체 파이프라인이 이 QA 생성을 위해 존재합니다.
